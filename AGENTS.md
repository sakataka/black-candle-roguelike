# AGENTS.md

このリポジトリでは、回答・作業報告は日本語で行う。

## プロジェクト方針

- 目的は NetHack 互換ではなく、TypeScript 製の独自ローグライクを作ること。
- 既存の NetHack ソースや旧ブリッジ実装は、必要がない限り参照・流用しない。
- 設計素材として主に `docs/roguelike-role-candidates.md`、`docs/roguelike-enemy-candidates.md`、`docs/roguelike-item-equipment-candidates.md`、`docs/roguelike-encounter-room-candidates.md` を使う。
- NetHack 由来データは seed として扱い、名前、世界観、数値、UI文言は新作用に作り直す。
- 世界観は暗めの王道ファンタジーを基本にする。

## アーキテクチャ

- ゲーム状態とルールは `src/game/core/` に置き、PixiJSやDOMに依存させない。
- 表示は `src/game/renderer/`、AIは `src/game/ai/`、コンテンツ定義は `src/game/content/` に分ける。
- 人間操作とAIは同じ `GameAction` を使う。
- AI/renderer向けの状態は `GameObservation` 経由で渡し、将来のLLM操作やterminal rendererを追加しやすくする。
- 設定や敵・アイテム・地形はなるべくデータ定義に寄せ、ゲームロジックへ直接埋め込まない。

## PDCA/PDCM とチューニング

- ゲーム機能、敵、アイテム、職業、AI、マップ生成、戦闘ルールを追加・変更したら、単発確認で終えず、`simulate:batch` で複数seed・複数職業の結果を比較してチューニングする。
- 標準の確認は `bun run simulate:batch -- --seeds 20260504:20260533 --turns 1600 --roles all --label baseline` を基準にする。CPU負荷を抑えたい場合は `--jobs 4`、再現性確認を優先する場合は `--jobs 1` を使う。デフォルトは `--jobs 8`。
- 変更案を試す時は、まず `public/config/game-balance.json` を `tmp/balance/*.json` にコピーして候補configを作り、`--config baseline=public/config/game-balance.json --config candidate=tmp/balance/candidate.json` のように同じseed集合で比較する。
- 評価では勝率だけでなく、到達階、死亡原因、低HPターン、探索停滞、既知罠踏み、拾得、戦闘、階層移動、アイテム使用、実行時間を確認する。
- チューニングしにくい数値・閾値・重み・出現率・AI判断条件がソースコード内に固定されている場合は、可能な範囲で `public/config/game-balance.json` やコンテンツ定義へ外出しし、batch比較で調整できる形へ徐々に直す。
- AIロジックも、完全なコード分岐に閉じ込めず、回復閾値、探索重み、危険回避、アイテム使用優先度などの調整可能な部分は設定値・データ定義へ寄せる。AI変更後は人間プレイヤーと同じ `GameObservation` 前提が保たれているか確認する。
- batch結果は `tmp/sim-reports/` に出す。`tmp/` はGit管理外なので、レポートを恒久資料にしたい場合だけ要点をREADMEやdocsへ反映する。

## 視界と探索

- 一度見えた地形は `explored` として残し、未探索の黒とは明確に区別する。
- 床・壁・階段は「現在視界」と「探索済み記憶」で見た目を分ける。
- プレイヤー、敵、アイテムの表示ルールを変える場合は、人間プレイヤーとAIが同じ前提で判断できるか確認する。
- アイテムは探索済みの場所なら表示に残してよい。敵は原則として現在視界内だけを確定情報として扱う。

## 画像・アセット運用

- SVG個別ファイルではなく、生成AIで作ったPNGスプライトシートを基本にする。
- 地形はタイルとして並べる前提で、枠やカード状の縁が出ない画像にする。
- 床・壁は大量に並べても違和感が出にくい反復向けの模様にする。床は歩行可能面として控えめでやや明るく、壁は床と一目で区別できる暗さ・厚み・輪郭を持たせる。
- 未探索と探索済み地形はローグライクの根幹なので、見た目で明確に区別する。階層テーマや地形画像を変える時もこの区別を壊さない。
- 視界状態の分かりにくさを、主人公やカーソルの強調で代替しない。マップタイル自体で未探索と探索済みを判別できるようにする。
- 視界は壁に遮られるFOVを基本にし、未探索地形は表示しない。一度探索した地形は通常表示のまま地図として残し、アイテム・罠・イベントも発見後に通常表示で残す。敵は原則として現在視界内だけ表示する。
- マップ生成は単なるランダム塗りではなく、部屋・通路・階段までの探索導線が感じられる構成を優先する。
- 主人公、敵、アイテムは背景透過PNGにする。クロマキー生成した場合は透過処理してから使う。
- 主人公や職業キャラクターは床と明度・色相・シルエットが被らないようにし、職業ごとの差が64px表示でも分かる見た目にする。
- 罠や地面ギミックは既存の床へ重ねる前提で、床タイルや背景を含めず、ギミック本体だけを透過PNGにする。
- 画像は `public/assets/sprites/` に保存し、`assetCatalog` から `path` と `sheet` 情報で切り出す。
- 1画像1モンスターではなく、同系統の敵・アイテムをまとめたスプライトシートを作り、その中から切り出して使う。
- entity ID と asset ID は分離し、画像差し替えでゲームデータを壊さない。
- 新規敵・アイテム・武器・イベント追加時は、基本的に新規PNGスプライトシートを作成する。既存スプライト流用は一時検証に限り、完成扱いにしない。
- 手続き生成のテスト画像、仮アイコン、単純な図形塗り、クロマキー未処理画像を完成アセットとして使わない。追加コンテンツの完成扱いには、世界観に合う生成AIスプライトシートを作り、必要な透過処理と画面確認まで行う。

## コンテンツ追加

- 敵は `family + role + tier` の考え方で増やす。
- 新しい敵・アイテムを追加する時は、少なくとも以下を更新する。
  - `src/game/content/entities.ts`
  - `src/game/content/assets.ts`
  - 必要なら `src/game/core/game.ts` の出現テーブルや効果処理
- アイテム効果はまず小さく実装し、AIが扱えるように `GameAction` と `chooseAutoplayAction` も確認する。
- フロアごとに敵とアイテムが寂しくならないよう、出現数と種類を確認する。

## AI方針

- AIは決定的・再現可能なルールベースを基本にする。
- AIは人間と同じ表示/探索済み情報を前提に判断する。
- 優先順位は概ね、回復、装備更新、足元の拾得、隣接敵攻撃、見えているアイテム取得、見えている敵への対応、階段、探索。
- AI変更後はheadless simulationで、攻撃・拾得・階層移動が起きることを確認する。

## 検証

- 変更後は最低限 `bun run build` を実行する。
- AIやcore変更後は、headless simulationでクラッシュせず進行することを確認する。
- ゲーム性・バランス・AI判断・出現テーブル・職業・敵・アイテムに関わる変更後は、`simulate:batch` で複数seed/職業の到達分布を確認し、必要なら候補config比較でPDCA/PDCMを回す。
- 見た目を変えた場合は、起動中の `http://127.0.0.1:1420/` をブラウザでリロードしてスクリーンショット確認する。
- 既存dev serverが起動中なら、不要に別サーバーを増やさない。

## スコープ注意

- 旧NetHack/Tauri/xtermブリッジの復旧や互換対応に戻らない。
- 大きなリファクタより、ゲームとして遊べる改善を小さく積む。
- BGM、効果音、アニメーションは後回し。ただしasset catalogやrendererの設計では後から足せる余地を残す。
